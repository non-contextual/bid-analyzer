## 异步处理升级
**What:** 将 API 请求处理从同步改为异步任务队列。
**Why:** 当前 MVP 同步处理在 OCR 或大文件场景会阻塞连接，单 worker 只能同时服务一个用户。
**Pros:** 支持多用户并发，避免 HTTP 超时，提升用户体验。
**Cons:** 需要引入 Celery + Redis 或 FastAPI BackgroundTasks，增加部署复杂度。
**Context:** MVP 验证阶段（仅 3-5 个测试用户）不需要，等有付费用户后再做。从 FastAPI BackgroundTasks 开始，满足后再升级 Celery。
**Depends on:** 用户验证通过，有稳定用户量。

## 自建爬取层 / 数据摄入策略
**What:** 评估是否自建爬虫从政府采购平台抓取公告，或接入现有平台（千里马等）API。
**Why:** 当前用户需手动上传文件，限制了产品的自动化程度和用户规模。长期依赖第三方平台有被"功能化"（对方推出竞争分析层）的风险。
**Pros:** 数据自主，摆脱上游平台依赖，可做自动推送。
**Cons:** 爬虫维护成本高，法律灰色地带，现有大平台数据更全。
**Context:** 独立 AI 评审标记此项为"18 个月内需要重新评估的战略风险"。如果大平台在此期间推出分析功能，需要加速此计划。先通过 API 接入验证，再决定是否自建。
**Depends on:** 用户验证通过，有付费意愿，且大平台未推出直接竞争功能。
